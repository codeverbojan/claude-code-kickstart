# Learn Project Patterns

Analyze this codebase and generate (or update) `patterns.md` at the project root.
Do NOT ask questions — discover everything from the code. Do NOT write generic
advice — only document patterns that actually exist in this codebase.

## What to Extract

For each category below, search the codebase for real examples. If a pattern
doesn't exist in this project, skip that category entirely.

### 1. Error Handling
- How are errors created? (custom error classes, error factories, Result types)
- How are errors caught? (try/catch, middleware, error boundaries)
- Where is the error handler? (file path + pattern)
- Example: "All API routes use try/catch → AppError → errorHandler middleware"

### 2. Test Structure
- Where do tests live? (co-located, `__tests__/`, `tests/`, `test/`)
- What framework? (vitest, jest, pytest, go test)
- Fixtures/mocks: where and how?
- Is the DB mocked or real in tests?
- Example: "Tests mirror src/. Fixtures in __fixtures__/. Real test DB, no mocks."

### 3. Import / Module Patterns
- Path aliases? (check tsconfig.json paths, pyproject.toml, go.mod)
- Barrel exports? (index.ts files that re-export)
- Circular dependency patterns to avoid?
- Example: "@/ maps to src/. Each module has index.ts barrel export."

### 4. API / Route Patterns
- Route file structure (file-based routing, controller pattern, etc.)
- Middleware chain (auth → validate → handler → error)
- Request/response types (Zod schemas, Pydantic, typed structs)
- Example: "app/api/[resource]/route.ts. Auth middleware in middleware.ts."

### 5. Component Patterns (if frontend)
- Server vs client component split
- Styling approach (CSS Modules, Tailwind, styled-components)
- State management (context, zustand, redux)
- Naming conventions
- Example: "Server components by default. 'use client' only in interactive/"

### 6. Database Patterns
- ORM/query builder (Drizzle, Prisma, SQLAlchemy, sqlx)
- Migration approach
- Naming conventions (snake_case tables, camelCase fields?)
- Example: "Drizzle ORM. Migrations in packages/db/migrations/."

### 7. Naming Conventions
- File naming (kebab-case, PascalCase, snake_case)
- Variable/function naming
- Component naming
- Branch/commit patterns
- Discover by reading 10+ files, not by guessing

### 8. Project-Specific Gotchas
- Anything unusual or non-obvious about how this project works
- Things that would trip up someone new to the codebase
- Workarounds currently in place

## How to Search

1. Read the top-level file structure
2. Read config files: tsconfig.json, pyproject.toml, go.mod, Cargo.toml, etc.
3. Read 3-5 representative source files across different modules
4. Read 2-3 test files to understand testing patterns
5. Read middleware/error handling files
6. Grep for patterns: `import`, `throw`, `catch`, `describe`, `test(`, etc.

## Rules

- Every claim must reference a real file path
- Only document patterns that exist in 2+ files (not one-offs)
- If you find conflicting patterns, note both and flag the inconsistency
- Keep it concise (aim for 50-100 lines) — this is loaded at session start
- If space is tight, prioritize: Error Handling, Test Structure, Naming Conventions
- If patterns.md already exists, update it rather than overwriting blindly:
  - Sections starting with `## ` followed by auto-generated content can be replaced
  - Any section with `<!-- manual -->` on the heading line must be preserved as-is
  - When in doubt, preserve existing content and append new findings

## After Writing

Report what you found and what surprised you. Flag any inconsistencies
or anti-patterns worth discussing.
